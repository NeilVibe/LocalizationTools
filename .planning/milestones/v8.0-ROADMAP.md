# Roadmap: LocaNext v8.0 Service Layer Extraction (ARCH-03)

## Overview

Extract business logic from 6 thick FastAPI route handlers (4,032 lines) into proper service classes. Creates route→service→repository architecture. Eliminates ~60% duplication in stats/rankings. Enables unit testing. Zero user-facing changes — API contracts stay identical.

## Phase Details

### Phase 69: Stats & Rankings Service
**Goal**: All stats/rankings query logic lives in service classes, route handlers are ≤10 lines, and shared date-range helper eliminates massive duplication
**Depends on**: Nothing (first phase — highest duplication, highest impact)
**Requirements**: SVC-STATS-01, SVC-STATS-02, SVC-STATS-03, SVC-STATS-04
**Success Criteria**:
  1. `server/services/stats_service.py` contains all 14 query builders from stats.py
  2. `server/services/rankings_service.py` contains all 6 query builders from rankings.py
  3. Date-range calculation is a single shared method, not duplicated 20 times
  4. No N+1 queries — top_tool_per_user uses window function instead of loop
  5. All stats/rankings endpoints return identical JSON responses as before (regression-safe)

### Phase 70: Auth Service
**Goal**: Auth logic is unit-testable without HTTP — login, register, password change, rate limiting, audit logging all encapsulated in AuthService
**Depends on**: Nothing (independent)
**Requirements**: SVC-AUTH-01, SVC-AUTH-02, SVC-AUTH-03
**Success Criteria**:
  1. `server/services/auth_service.py` handles all auth business logic
  2. Route handlers in auth_async.py are ≤10 lines each
  3. Rate limiting, password hashing, token creation are service methods
  4. Existing login/register flow works identically (regression-safe)

### Phase 71: Telemetry & Remote Logging Services
**Goal**: Telemetry queries and log ingestion pipeline extracted into services, N+1 queries fixed
**Depends on**: Nothing (independent)
**Requirements**: SVC-TELEM-01, SVC-TELEM-02, SVC-TELEM-03
**Success Criteria**:
  1. `server/services/telemetry_service.py` contains all admin_telemetry query logic
  2. `server/services/remote_logging_service.py` contains log ingestion + session tracking
  3. Installations list uses JOIN instead of N+1 per-installation queries
  4. All telemetry/logging endpoints return identical responses

### Phase 72: Infrastructure Services
**Goal**: DB health, system health, and progress operations separated into service layer
**Depends on**: Nothing (independent)
**Requirements**: SVC-INFRA-01, SVC-INFRA-02, SVC-INFRA-03
**Success Criteria**:
  1. Raw SQL in admin_db_stats.py encapsulated in `server/services/db_health_service.py`
  2. Health helper functions from health.py moved to `server/services/health_service.py`
  3. WebSocket dispatch in progress_operations.py separated from CRUD logic
  4. All endpoints return identical responses
