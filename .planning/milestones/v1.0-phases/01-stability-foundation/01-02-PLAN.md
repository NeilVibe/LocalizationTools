For each of the 5 repository files, create a test file following this pattern:

**General pattern for all test files:**
- `from __future__ import annotations` at top
- `import pytest` and `@pytest.mark.stability` on each test
- Each test function takes `db_mode` and the relevant repo fixture from conftest.py
- Tests call the repository method, then assert results using `assert_equivalent` from conftest
- For CRUD operations: create -> get -> verify fields match -> update -> verify update -> delete -> verify gone
- Use game-realistic data (Korean/English game strings, realistic field values)
- Naming convention: `test_{repo}_{method}` (pytest parametrize adds `[online]`/`[server_local]`/`[offline]` automatically)

**Read each interface file** (`server/repositories/interfaces/{name}_repository.py`) to get the exact method signatures and parameters before writing tests.

**test_platform_repo.py** (~13 methods):
- test_platform_create: Create platform with name="PC", verify returned dict has name, owner_id
- test_platform_get: Create then get by ID, verify fields match
- test_platform_get_all: Create 2 platforms, get_all returns both
- test_platform_update: Create, update name to "Console", verify
- test_platform_delete: Create, delete, get returns None
- test_platform_get_with_project_count: Create platform + project, verify count=1
- test_platform_set_restriction: Create, restrict, verify is_restricted=True
- test_platform_check_name_exists: Create "PC", check_name_exists("PC")=True, check_name_exists("Xbox")=False
- test_platform_count: Create 3, count()=3
- test_platform_get_projects: Create platform + 2 projects, verify list
- test_platform_search: Create platforms, search by name substring
- test_platform_get_accessible: Verify returns list (may differ between modes -- document differences)

**test_project_repo.py** (~10 methods): Similar CRUD + platform assignment tests

**test_folder_repo.py** (~8 methods): CRUD + tree operations (nested folders)

**test_file_repo.py** (~14 methods): CRUD + search + move/copy. Files need project_id and folder_id from fixture hierarchy

**test_row_repo.py** (~12 methods): CRUD + bulk_create + history. Rows need file_id. Use Korean source strings ("새 게임", "계속하기") and English targets. Test bulk operations with 3+ rows.

**Key pitfalls to handle (from research):**
- Boolean fields: normalize before comparison (SQLite returns 0/1)
- IDs: use assert_equivalent which ignores "id" field
- Timestamps: ignored by assert_equivalent
- Some methods may not exist in SQLite repos (they should raise or return defaults) -- use `pytest.skip` if method genuinely not applicable to a mode, or `pytest.xfail` if it's a known gap

**CapabilityRepository asymmetry:** The capability repo is a stub in SQLite modes. Its tests go in Task 2 with appropriate expectations.
