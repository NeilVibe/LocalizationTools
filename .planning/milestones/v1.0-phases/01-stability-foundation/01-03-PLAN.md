**Part A — Pre-startup port cleanup (server/main.py):**
Read `server/main.py` fully to understand current startup flow. Add a port cleanup step at the VERY BEGINNING of the startup sequence (before Uvicorn binds):

- Use `psutil` to check if port 8888 is already in use
- If occupied: log a WARNING (not ERROR) with the PID, then kill the stale process
- This prevents "Address already in use" failures from prior crashes
- Only do this in DEV_MODE to avoid killing production processes
- Implementation: add a `_cleanup_stale_port(port: int)` function, call it in the lifespan startup

**Part B — Startup reliability test (test_startup.py):**
Create `tests/stability/test_startup.py` with:

1. **Helper: `_cleanup_port(port: int)`**: Use psutil to find and kill any process on the given port. Wait up to 2s for it to die.

2. **Helper: `_start_server(**extra_env) -> subprocess.Popen`**: Start `python3 server/main.py` with env `DEV_MODE=true`, `DATABASE_MODE=sqlite` (use SQLite to avoid PostgreSQL dependency for startup tests), plus any extra_env overrides. Return the Popen object. Capture stdout and stderr.

3. **Helper: `_wait_for_health(timeout_s: float = 5.0) -> float`**: Poll `http://127.0.0.1:8888/health` every 100ms until success or timeout. Return elapsed time. Raise on timeout.

4. **`test_dev_startup_10_consecutive_times()`** (STAB-01):
   - Loop 10 times:
     - Call `_cleanup_port(8888)` first
     - Start server via `_start_server()`
     - Wait for health check, measure elapsed time
     - Assert elapsed < 5.0 seconds
     - Read stderr output, assert "ERROR" not in it (zero ERROR log lines)
     - Terminate server, wait for exit
     - Call `_cleanup_port(8888)` again
   - All 10 must succeed. Fail fast on first failure with attempt number.

5. **`test_startup_db_connectivity_check()`** (STAB-01):
   - Start server in SQLite mode
   - Hit /health endpoint
   - Verify response includes DB connectivity status (if health endpoint reports it)
   - If not, verify server started without DB-related errors in logs

6. **`test_startup_clear_error_on_port_conflict()`** (STAB-01):
   - Start first server instance
   - Wait for health
   - Start second server instance on same port
   - Verify second instance either: (a) kills the first and takes over (due to pre-startup cleanup), or (b) fails with a clear error message — NOT a cryptic traceback
   - Clean up both processes

7. **`test_startup_rejects_weak_secret_key()`** (STAB-01, per user decision: "Strict security validation"):
   - Start server with env `SECURITY_MODE=strict` and `SECRET_KEY=weak` (a short, weak key)
   - Assert the process exits with non-zero exit code (SystemExit from config.py validation)
   - Assert stderr/stdout contains a clear error message about SECRET_KEY (e.g., "SECRET_KEY" and "Security validation FAILED")
   - Also test with the default key: start server with `SECURITY_MODE=strict` and NO SECRET_KEY override (so it uses "dev-secret-key-CHANGE-IN-PRODUCTION")
   - Assert same non-zero exit + clear error behavior
   - Verify port 8888 is NOT occupied after the rejected startup (server did not partially start)
   - Note: server/config.py already has this logic (`SECURITY_MODE == "strict"` raises SystemExit). This test VERIFIES it works end-to-end as a subprocess.

Mark all tests with `@pytest.mark.stability`. Use `@pytest.mark.timeout(120)` for the 10-start test (pytest-timeout).

Use `from __future__ import annotations` at top.
