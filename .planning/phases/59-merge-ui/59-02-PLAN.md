```javascript
function openMerge() {
  if (!$selectedProject) {
    logger.warn("No project selected — cannot open merge modal");
    return;
  }
  logger.userAction("Merge modal opened", { projectId: $selectedProject.id });
  mergeMultiLanguage = false;
  mergeFolderPath = '';
  showMergeModal = true;

function openMergeFolder(folderPath) {
  if (!$selectedProject) return;
  logger.userAction("Merge folder modal opened", { projectId: $selectedProject.id, folderPath });
  mergeMultiLanguage = true;
  mergeFolderPath = folderPath;
  showMergeModal = true;
}
```

**6. Pass openMergeFolder callback to LDM app:**
Find where FilesPage is rendered (it is inside LDM.svelte which is in +layout.svelte's Content area). The callback chain is: +layout.svelte -> LDM.svelte -> FilesPage.svelte. Since FilesPage takes callback props, we need to thread the merge callback.

HOWEVER: Looking at the architecture, FilesPage is NOT rendered directly in +layout.svelte. It's rendered inside LDM.svelte. The simplest approach: use a store-based signal or window event. BUT the project pattern uses callback props (not stores for actions).

BEST APPROACH: Use a custom event on window. In +layout.svelte, listen for it:
```javascript
import { onMount } from "svelte"; // Already imported
// Inside onMount (or $effect):
$effect(() => {
  function handleMergeFolderEvent(e) {
    openMergeFolder(e.detail.folderPath);
  }
  window.addEventListener('merge-folder-to-locdev', handleMergeFolderEvent);
  return () => window.removeEventListener('merge-folder-to-locdev', handleMergeFolderEvent);
});
```
This avoids threading a callback through 3 layers of components.
